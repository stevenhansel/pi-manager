# Configuration Architecture

A design pattern where reusable resources live in a global **pool** and each profile is a directory containing a lightweight JSON **manifest** that selects from the pool and declares profile-specific configuration, while serving directly as the active agent directory.

## Core concepts

```
~/.pim/
├── pool/                ← SOURCE OF TRUTH: the actual files
│   ├── extensions/
│   ├── skills/
│   ├── prompts/
│   └── models/          ← model provider definitions
│
├── profiles/            ← ISOLATED PROFILE DIRECTORIES: one per profile
│   ├── work/            ← PI_CODING_AGENT_DIR = this directory
│   │   ├── manifest.json    ← SELECTION: lightweight JSON manifest
│   │   ├── settings.json    ← generated from settings key in manifest
│   │   ├── mcp.json         ← generated from mcp keys in manifest
│   │   ├── models.json      ← generated from pool model selections
│   │   ├── config/          ← seeded defaults + manifest config overrides
│   │   ├── sessions/        ← pi session history (persistent)
│   │   ├── auth.json        ← login tokens
│   │   ├── trust.json       ← trust decisions
│   │   └── APPEND_SYSTEM.md ← system prompt append
│   │
│   └── cad/
│       └── ...
│
└── pim.json             ← stores default profile name & global configuration
```

### Pool — the source of truth

The `pool/` directory holds all reusable resources. These are the actual files — extensions, skills, prompts — that you develop and maintain. The pool is profile-agnostic: it doesn't care which profiles use which subset.

```
pool/
├── extensions/
│   ├── rtk.ts                     ← a single-file extension
│   ├── searxng/                   ← a multi-file extension
│   │   ├── index.ts
│   │   └── README.md
│   └── homeassistant-mcp/
├── skills/
│   ├── web-research/
│   │   └── SKILL.md
│   └── freecad/
├── prompts/
│   ├── review.md
│   └── commit.md
└── models/
    ├── anthropic/                 ← model provider directory
    │   └── model.json
    └── openai/
        └── model.json
```

### Manifest — the selection

A profile manifest is a single `manifest.json` file inside the profile directory that says *what* to pull from the pool and *how* to configure things. It never duplicates file contents — it only references them by name. The `settings.defaultModel` and `settings.defaultProvider` keys control which model pi selects by default.

```json
{
  "select": {
    "extensions": ["rtk", "searxng"],
    "skills": ["web-research"],
    "prompts": ["review", "commit"],
    "models": ["anthropic", "openai"]
  },
  "settings": {
    "defaultProvider": "anthropic",
    "defaultModel": "claude-sonnet-4-6",
    "theme": "dark"
  },
  "mcpServers": {
    "homeassistant": {
      "command": "npx",
      "args": ["-y", "@home-assistant/mcp"],
      "env": ["HA_TOKEN", "HA_URL"]
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "."]
    }
  }
}
```

Key principle: **a manifest only contains references and configuration, never file contents.**

### Profile Directory — the Consolidated State

The profile directory holds both the manifest and the runtime state that is auto-generated by pi — auth tokens, session logs, trust decisions, and config files. This is never version-controlled and never hand-edited.

```
profiles/work/
├── config/          ← config files (e.g. searxng.json), written once then persisted
├── sessions/        ← pi session history
├── auth.json        ← written by pi when you /login
├── models.json      ← model configuration
├── trust.json       ← project trust decisions
└── APPEND_SYSTEM.md ← system prompt append
```

Seeded config files are created from the manifest's `configs` field or the pool template's defaults on profile activation, then preserved across rebuilds.

## How pi finds its config (`PI_CODING_AGENT_DIR`)

Pi reads the `PI_CODING_AGENT_DIR` environment variable on startup. If set,
it uses that directory instead of `~/.pi/agent`. pim sets this variable to
the profile's directory before launching pi.

This is the key architectural insight: **pim never modifies `~/.pi/agent`**.
Instead, it launches a new pi process with the right environment variable.
This means:

- Multiple pi instances can run simultaneously with different profiles
- The original `~/.pi/agent` (if it exists) is left untouched
- No risk of crashing a running pi by changing a global symlink

```bash
# What pim does internally:
PI_CODING_AGENT_DIR=~/.pim/profiles/research exec pi

# You can also use this directly:
PI_CODING_AGENT_DIR=~/.pim/profiles/research pi -p "hello"
```

## Activation / Set-Default Flow

When `pim set-default <name>` runs (or `pim <name>` runs for a profile that has not been initialized yet), it builds/refreshes the profile directory at `~/.pim/profiles/<name>/` and (for `pim <name>`) exec's into `pi` with `PI_CODING_AGENT_DIR` set to that directory:

```
~/.pim/profiles/<name>/        ← NOT symlinked from ~/.pi/agent
├── manifest.json                     ← selection manifest
├── settings.json                     ← from profile manifest
├── mcp.json                          ← from profile manifest
├── config/
│   └── searxng.json                  ← config override or seeded from pool template
├── extensions/
│   ├── rtk.ts           ── symlink → pool/extensions/rtk.ts
│   └── searxng/         ── symlink → pool/extensions/searxng/
├── skills/
│   └── web-research/    ── symlink → pool/skills/web-research/
├── prompts/
│   ├── review.md        ── symlink → pool/prompts/review.md
│   └── commit.md        ── symlink → pool/prompts/commit.md
├── auth.json                         ← persistent auth state
├── models.json                       ← persistent models configuration
├── trust.json                        ← persistent trust configuration
└── sessions/                         ← persistent session logs
```

1. Read the profile manifest from `profiles/<name>/manifest.json`
2. Resolve each selected item from the pool
3. Create symlinks for selected items into the profile directory
4. Write `settings` and `mcpServers` into `settings.json` and `mcp.json` respectively
5. Sync config files from manifest to `config/` (seeding default templates and applying custom overrides)
6. Set profile as default (for `pim set-default`) or exec into pi (for launcher mode)

**Key difference from the old design:** `~/.pi/agent` is never modified. pi
reads its config from `$PI_CODING_AGENT_DIR`, not `~/.pi/agent`.

## What goes where

| Category | Location | Example | Why |
|----------|----------|---------|-----|
| Extensions | `pool/extensions/` | `rtk.ts`, `searxng/` | Shared, reusable, developed independent of any profile |
| Skills | `pool/skills/` | `web-research/` | Same as extensions |
| Prompts | `pool/prompts/` | `review.md` | Same — reusable templates shared across profiles |
| Settings | Profile manifest | `settings.defaultProvider` | Per-profile preferences (different models, theme per context) |
| MCP servers | Profile manifest | `mcpServers.filesystem` | Environment-specific — each profile needs different servers |
| Models | `pool/models/` | `model.json` per entry | Provider + model definitions selected per profile, generated into `models.json` |
| Config files | `profiles/<name>/config/` | `searxng.json` | Per-profile config persisted across rebuilds |
| Auth tokens | `profiles/<name>/` | `auth.json` | Auto-generated secrets, never version-controlled |
| Models | `pool/models/` + profile build | `models.json` | AI provider/model definitions in pool; generated per profile from selections |
| Model providers | `pool/models/` | `model.json` | Reusable provider+model entries; selected per profile, built into `models.json` |
| Sessions | `profiles/<name>/` | `sessions/` | Runtime state, persisted across rebuilds |
| Trust decisions | `profiles/<name>/` | `trust.json` | Auto-generated, machine-local |
| System prompt | `profiles/<name>/` | `APPEND_SYSTEM.md` | Per-profile system prompt append |

## Pool Models

Model provider definitions live in `pool/models/<name>/model.json`. Each entry defines a provider configuration and its available models, following pi's `models.json` format. Selected models are built into the profile's `models.json` on activation.

### Pool Model Entry

```json
{
  "name": "Anthropic",
  "description": "Anthropic Claude models — Sonnet 4, Opus 4",
  "tags": ["model", "anthropic"],
  "provider": "anthropic",
  "api": "anthropic-messages",
  "apiKey": "$ANTHROPIC_API_KEY",
  "config_fields": [
    {
      "key": "apiKey",
      "label": "Anthropic API Key",
      "type": "password",
      "required": true,
      "help": "Get your API key at https://console.anthropic.com/",
      "envVar": "ANTHROPIC_API_KEY",
      "envPriority": "prefer"
    }
  ],
  "models": [
    {
      "id": "claude-sonnet-4-20250514",
      "name": "Claude Sonnet 4",
      "reasoning": true,
      "input": ["text", "image"],
      "contextWindow": 200000,
      "maxTokens": 64000,
      "cost": { "input": 3, "output": 15, "cacheRead": 0.3, "cacheWrite": 3.75 }
    }
  ]
}
```

- `provider` — the key used in the generated `providers` object (e.g., `"anthropic"`, `"openrouter"`)
- `api` — pi API type: `"anthropic-messages"`, `"openai-completions"`, `"openai-responses"`, `"google-generative-ai"`
- `apiKey` — supports pi's value resolution (`$ENV_VAR`, `!command`, literals)
- `config_fields` — TUI metadata: fields shown in the configure view and used for template substitution
- `models` — array of model definitions following pi's [model configuration format](models.md)

### Backward Compat: Inline Providers

For quick one-off model configs, use `modelsProviders` in the manifest:

```json
{
  "select": { "models": [] },
  "modelsProviders": {
    "ollama": {
      "baseUrl": "http://localhost:11434/v1",
      "api": "openai-completions",
      "apiKey": "ollama",
      "models": [{ "id": "llama3.1:8b" }]
    }
  }
}
```

Inline providers are only used when no pool selection with the same key exists.

### Generated Output

The build step generates `models.json` in the profile directory:

```json
{
  "providers": {
    "anthropic": {
      "api": "anthropic-messages",
      "apiKey": "$ANTHROPIC_API_KEY",
      "models": [...]
    }
  }
}
```

This is the file pi reads at runtime. You can also edit it manually — pi reloads model config on each `/model` invocation.

### Suppressing Built-in Models

pi ships with built-in providers (Anthropic, OpenAI, DeepSeek, OpenRouter, Google, etc.). To restrict which built-in models appear, use the `enabledModels` setting in the manifest:

```json
{
  "settings": {
    "defaultModel": "claude-sonnet-4-20250514",
    "defaultProvider": "anthropic",
    "enabledModels": ["claude-*", "gpt-*"]
  }
}
```

This is a glob pattern filter that controls which models appear in the Ctrl+P model cycler.

## Benefits over the old approach

| Aspect | Old approach (copies) | Pool Manifest |
|--------|----------------------|---------------|
| **Development loop** | Edit copy, then re-copy to other profiles | Edit once in pool, all profiles see it |
| **Profile size** | Full directories, potentially MBs | Directory containing manifest + symlinks & state |
| **Diffing profiles** | Deep directory comparisons | `diff profiles/work/manifest.json profiles/cad/manifest.json` |
| **Adding a new profile** | Copies everything | Create a directory with a 10-line `manifest.json` |
| **Adding a new extension** | Drop it in each profile | Drop it in pool, reference in manifest |
| **Version control** | Can't track what changed in profiles | JSON is trivially diffable |
| **Deleting a profile** | May leave unused copies | No data loss — pool is untouched |

## Prior art

This pattern draws from:

- **NixOS / Nixpkgs** — global package pool, `configuration.nix` selects which modules to enable
- **VS Code** — marketplace pool, `settings.json` enables extensions
- **Spacemacs** — standard layer pool, `.spacemacs` selects layers
- **Codex CLI** — flat top-level defaults, per-profile overrides
- **CLI Configuration Management Patterns** — layered hierarchy of defaults → file → env → flags
