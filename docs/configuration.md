# Configuration Architecture

A design pattern where reusable resources live in a global **pool** and each profile is a lightweight JSON **manifest** that selects from the pool and declares profile-specific configuration.

## Core concepts

```
~/.pi-manager/
├── pool/                ← SOURCE OF TRUTH: the actual files
│   ├── extensions/
│   ├── skills/
│   └── prompts/
│
├── profiles/            ← SELECTION: lightweight JSON manifests
│   ├── work.json
│   └── cad.json
│
├── data/                ← STATE: auto-generated, profile-specific
│   ├── work/
│   │   ├── config/          ← per-profile config files (persistent)
│   │   ├── sessions/        ← pi session history
│   │   ├── auth.json        ← login tokens
│   │   ├── models.json      ← model configuration
│   │   ├── trust.json       ← trust decisions
│   │   └── APPEND_SYSTEM.md ← system prompt append
│   └── cad/
│       └── ...
│
├── .active/             ← BUILT VIEWS: one per profile, used by pi
│   ├── work/            ← PI_CODING_AGENT_DIR = this directory
│   └── cad/
│
└── default              ← stores the default profile name
```

### Pool — the source of truth

The `pool/` directory holds all reusable resources. These are the actual files — extensions, skills, prompts — that you develop and maintain. The pool is profile-agnostic: it doesn't care which profiles use which subset.

```
pool/
├── extensions/
│   ├── rtk.ts                     ← a single-file extension
│   ├── searxng/                   ← a multi-file extension
│   │   ├── index.ts
│   │   └── README.md
│   └── homeassistant-mcp/
├── skills/
│   ├── web-research/
│   │   └── SKILL.md
│   └── freecad/
└── prompts/
    ├── review.md
    └── commit.md
```

### Manifest — the selection

A profile is a single JSON file that says *what* to pull from the pool and *how* to configure things. It never duplicates file contents — it only references them by name.

```json
{
  "select": {
    "extensions": ["rtk", "searxng"],
    "skills": ["web-research"],
    "prompts": ["review", "commit"]
  },
  "settings": {
    "defaultProvider": "anthropic",
    "defaultModel": "claude-sonnet-4-6",
    "theme": "dark"
  },
  "mcpServers": {
    "homeassistant": {
      "command": "npx",
      "args": ["-y", "@home-assistant/mcp"],
      "env": ["HA_TOKEN", "HA_URL"]
    },
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "."]
    }
  }
}
```

Key principle: **a manifest only contains references and configuration, never file contents.**

### Data — the state

The `data/` directory holds runtime state that is auto-generated by pi —
auth tokens, session logs, trust decisions, and config files. This is
never version-controlled and never hand-edited.

```
data/work/
├── config/          ← config files (e.g. searxng.json), written once then persisted
├── sessions/        ← pi session history, synced from active view
├── auth.json        ← written by pi when you /login
├── models.json      ← model configuration
├── trust.json       ← project trust decisions
└── APPEND_SYSTEM.md ← system prompt append
```

The data directory is created lazily when the profile is first activated.
Config files are seeded from the manifest's `configs` field on first
activation, then preserved across rebuilds via symlinks.

## How pi finds its config (`PI_CODING_AGENT_DIR`)

Pi reads the `PI_CODING_AGENT_DIR` environment variable on startup. If set,
it uses that directory instead of `~/.pi/agent`. pim sets this variable to
the profile's active view before launching pi.

This is the key architectural insight: **pim never modifies `~/.pi/agent`**.
Instead, it launches a new pi process with the right environment variable.
This means:

- Multiple pi instances can run simultaneously with different profiles
- The original `~/.pi/agent` (if it exists) is left untouched
- No risk of crashing a running pi by changing a global symlink

```bash
# What pim does internally:
PI_CODING_AGENT_DIR=~/.pi-manager/.active/research exec pi

# You can also use this directly:
PI_CODING_AGENT_DIR=~/.pi-manager/.active/research pi -p "hello"
```

## Activation flow

When `pim <name>` runs (or `pim use <name>`), it builds the active view at
`~/.pi-manager/.active/<name>/` and (for `pim <name>`) exec's into `pi` with
`PI_CODING_AGENT_DIR` set to that directory:

```
~/.pi-manager/.active/<name>/        ← NOT symlinked from ~/.pi/agent
├── settings.json                     ← from profile manifest
├── mcp.json                          ← from profile manifest
├── config/
│   └── searxng.json   ── symlink  → data/<name>/config/searxng.json  (persistent)
├── extensions/
│   ├── rtk.ts           ── symlink → pool/extensions/rtk.ts
│   └── searxng/         ── symlink → pool/extensions/searxng/
├── skills/
│   └── web-research/    ── symlink → pool/skills/web-research/
├── prompts/
│   ├── review.md        ── symlink → pool/prompts/review.md
│   └── commit.md        ── symlink → pool/prompts/commit.md
├── auth.json            ── symlink  → data/<name>/auth.json
├── models.json          ── symlink  → data/<name>/models.json
├── trust.json           ── symlink  → data/<name>/trust.json
└── sessions/
    └── *.jsonl          ── symlink  → data/<name>/sessions/*.jsonl  (persistent)
```

1. Read the profile manifest from `profiles/<name>.json`
2. Resolve each selected item from the pool
3. Create symlinks for selected items into the active directory
4. Write `settings` and `mcpServers` into the appropriate config files
5. Sync config files from manifest to `data/<name>/config/` (seeding)
6. Link in profile-specific state (auth, sessions, configs) from `data/<name>/`
7. Set profile as default (for `pim use`) or exec into pi (for launcher mode)

**Key difference from the old design:** `~/.pi/agent` is never modified. pi
reads its config from `$PI_CODING_AGENT_DIR`, not `~/.pi/agent`.

## What goes where

| Category | Location | Example | Why |
|----------|----------|---------|-----|
| Extensions | `pool/extensions/` | `rtk.ts`, `searxng/` | Shared, reusable, developed independent of any profile |
| Skills | `pool/skills/` | `web-research/` | Same as extensions |
| Prompts | `pool/prompts/` | `review.md` | Same — reusable templates shared across profiles |
| Settings | Profile manifest | `settings.defaultProvider` | Per-profile preferences (different models, theme per context) |
| MCP servers | Profile manifest | `mcpServers.filesystem` | Environment-specific — each profile needs different servers |
| Config files | `data/<name>/config/` | `searxng.json` | Per-profile config persisted across rebuilds |
| Auth tokens | `data/<name>/` | `auth.json` | Auto-generated secrets, never version-controlled |
| Models | `data/<name>/` | `models.json` | AI provider/model configuration |
| Sessions | `data/<name>/` | `sessions/` | Runtime state, persisted across rebuilds |
| Trust decisions | `data/<name>/` | `trust.json` | Auto-generated, machine-local |
| System prompt | `data/<name>/` | `APPEND_SYSTEM.md` | Per-profile system prompt append |

## Benefits over the old approach

| Aspect | Old approach (copies) | Pool Manifest |
|--------|----------------------|---------------|
| **Development loop** | Edit copy, then re-copy to other profiles | Edit once in pool, all profiles see it |
| **Profile size** | Full directories, potentially MBs | Single JSON file, usually <1KB |
| **Diffing profiles** | Deep directory comparisons | `diff profiles/work.json profiles/cad.json` |
| **Adding a new profile** | `pim create --from` copies everything | Create a 10-line JSON file |
| **Adding a new extension** | Drop it in each profile | Drop it in pool, reference in manifest |
| **Version control** | Can't track what changed in profiles | JSON is trivially diffable |
| **Deleting a profile** | May leave unused copies | No data loss — pool is untouched |

## Inheritance (future consideration)

The manifest is self-contained, but in the future a profile could inherit from another:

```json
{
  "inherits": "default",
  "select": {
    "extensions": ["+heavy-tool"]     ← only the delta
  },
  "settings": {
    "defaultModel": "claude-haiku"    ← override one value
  }
}
```

At activation time, the child manifest would be deep-merged into the parent, with child values winning. This keeps manifests small while allowing shared base configurations.

## Prior art

This pattern draws from:

- **NixOS / Nixpkgs** — global package pool, `configuration.nix` selects which modules to enable
- **VS Code** — marketplace pool, `settings.json` enables extensions
- **Spacemacs** — standard layer pool, `.spacemacs` selects layers
- **Codex CLI** — flat top-level defaults, per-profile overrides
- **CLI Configuration Management Patterns** — layered hierarchy of defaults → file → env → flags
